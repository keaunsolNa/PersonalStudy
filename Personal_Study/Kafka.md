# Kafka 핵심 개념과 멘탈 모델

> 대상: Spring/Java 백엔드 엔지니어
> 목표: 카프카를 "메시지 큐"가 아니라 "분산 커밋 로그"로 이해하기

---

## 0. 핵심 정리

**Kafka는 메시지 큐가 아니라 분산 커밋 로그(distributed commit log)다.**

이 한 문장이 멘탈 모델의 전부다. 전통적인 메시지 큐(RabbitMQ, ActiveMQ)와의 결정적 차이를 먼저 잡으면 나머지 개념이 자연스럽게 연결된다.

| | 전통 MQ (RabbitMQ 등) | Kafka |
|---|---|---|
| 메시지 저장 | 소비되면 큐에서 **삭제** | 소비돼도 **로그에 보존** (보존 기간/용량 정책으로만 삭제) |
| 읽은 위치 관리 | **브로커**가 관리 (어디까지 줬는지) | **컨슈머**가 오프셋으로 관리 (어디까지 읽었는지) |
| 재처리 | 어려움 (메시지 이미 삭제됨) | 오프셋만 되감으면 **다시 읽기 가능** |
| 모델 | 푸시(push) 중심 | 풀(pull) 중심 |
| 비유 | 우체통 (꺼내면 사라짐) | 녹음된 방송 / append-only 파일 |

> **멘탈 모델 핵심:** 카프카의 토픽을 "끝에만 계속 추가되는(append-only) 거대한 로그 파일"이라고 상상하라. 프로듀서는 파일 끝에 줄을 추가하고, 컨슈머는 "내가 몇 번째 줄까지 읽었다"는 북마크(오프셋)를 들고 순서대로 읽어 내려간다.

---

## 1. Topic — 로그의 이름

**Topic**은 메시지가 쌓이는 논리적 카테고리다. DB로 치면 테이블, 파일 시스템으로 치면 디렉터리에 가깝다.

- 예: `order-created`, `payment-completed`, `user-signup`
- 프로듀서는 토픽에 쓰고, 컨슈머는 토픽에서 읽는다.
- 하나의 토픽은 여러 컨슈머가 **독립적으로** 읽을 수 있다 (한 명이 읽어도 메시지는 남아있으므로).

---

## 2. Partition — 병렬성과 순서의 단위 (가장 중요)

토픽은 물리적으로 하나의 로그가 아니라, 여러 개의 **Partition**으로 쪼개진다. 파티션이 카프카 이해의 핵심이다.

```text
Topic: order-created
┌───────────────────────────────────────────┐
│ Partition 0 │ [0][1][2][3][4][5] →         │
│ Partition 1 │ [0][1][2][3] →               │
│ Partition 2 │ [0][1][2][3][4][5][6][7] →   │
└───────────────────────────────────────────┘
                각 칸의 숫자 = Offset
```

파티션에 대해 반드시 기억할 4가지 사실:

1. **파티션은 병렬성의 단위다.** 컨슈머 그룹 내에서 파티션 하나는 동시에 한 컨슈머만 읽는다. 즉 **파티션 개수 = 최대 병렬 컨슈머 수**다. 파티션 3개면 컨슈머를 4개 띄워도 1개는 논다.
2. **순서는 파티션 안에서만 보장된다.** 한 파티션 내부의 메시지는 들어온 순서대로 읽힌다. 하지만 **서로 다른 파티션 간에는 순서 보장이 없다.** 토픽 전체의 전역 순서가 필요하면 파티션을 1개로 둬야 하는데, 그러면 병렬성이 사라진다 (트레이드오프).
3. **같은 Key는 같은 파티션으로 간다.** 메시지에 키를 지정하면 `hash(key) % partition수`로 파티션이 결정된다. 그래서 "같은 주문 ID의 이벤트는 항상 순서대로"가 필요하면 주문 ID를 키로 쓴다.
4. **파티션은 늘릴 수만 있고 줄일 수 없다.** 게다가 파티션 수를 바꾸면 키→파티션 매핑이 깨져서 순서 보장이 무너진다. 그래서 설계 단계에서 신중히 정해야 한다.

---

## 3. Offset — 컨슈머의 북마크

**Offset**은 파티션 안에서 각 메시지의 위치를 나타내는 0부터 시작하는 정수다.

- 파티션마다 독립적이다 (Partition 0의 offset 5와 Partition 1의 offset 5는 별개 메시지).
- 컨슈머는 "이 파티션을 offset N까지 처리했다"를 **커밋(commit)**해서 기록한다.
- 이 커밋된 오프셋은 `__consumer_offsets`라는 내부 토픽에 저장된다.

> **멘탈 모델:** 오프셋은 책의 페이지 번호이자 책갈피다. 컨슈머가 죽었다 살아나도 마지막 커밋한 오프셋부터 다시 읽으면 된다. 오프셋을 일부러 과거로 되감으면(reset) 처리했던 메시지를 다시 읽는 **재처리**가 가능하다 — 이게 카프카의 강력함이다.

**Consumer Lag**: `로그의 최신 오프셋 - 컨슈머가 커밋한 오프셋`. 컨슈머가 얼마나 밀려있는지를 나타내는 가장 중요한 운영 지표다. lag이 계속 커지면 컨슈머가 생산 속도를 못 따라가는 것이다.

---

## 4. Producer — 로그에 쓰는 쪽

**Producer**는 토픽으로 메시지를 보낸다. 핵심 동작:

- 메시지의 **Key로 파티션을 결정**한다 (키 없으면 라운드로빈/스티키 방식으로 분산).
- `acks` 설정으로 "몇 개의 복제본이 받았을 때 성공으로 칠지"를 정한다 — 신뢰성과 성능의 트레이드오프 (4단계에서 상세).
- 성능을 위해 메시지를 **배치(batch)**로 모아 보낸다.

---

## 5. Consumer & Consumer Group — 읽는 쪽

**Consumer**는 토픽에서 메시지를 읽는다. 그런데 카프카에서 컨슈머는 거의 항상 **Consumer Group** 단위로 동작한다. 이게 카프카 확장성의 핵심 메커니즘이다.

### 규칙

- **한 그룹 안에서**, 하나의 파티션은 정확히 하나의 컨슈머에게만 할당된다.
- 그룹 내 컨슈머들이 파티션을 **나눠 갖는다** → 부하 분산(scale-out).
- **서로 다른 그룹은 완전히 독립적이다** → 같은 토픽을 그룹 A는 결제 처리용으로, 그룹 B는 통계 집계용으로 각자 전체를 다 읽을 수 있다.

```text
Topic: order-created (Partition 0, 1, 2)

Consumer Group "payment-service"        Consumer Group "analytics-service"
┌──────────────────────────┐            ┌──────────────────────────┐
│ Consumer A ← Partition 0  │            │ Consumer X ← Partition 0  │
│ Consumer B ← Partition 1  │            │ Consumer X ← Partition 1  │
│ Consumer C ← Partition 2  │            │ Consumer Y ← Partition 2  │
└──────────────────────────┘            └──────────────────────────┘
   같은 메시지를 두 그룹이 각자 독립적으로 모두 소비한다
```

> **멘탈 모델:**
> - 같은 그룹 = "일을 나눠 하는 동료들" (한 메시지는 그룹 내 한 명만 처리)
> - 다른 그룹 = "같은 방송을 듣는 다른 청취자" (각자 전체를 다 들음)
>
> 전자가 큐(작업 분배), 후자가 발행-구독(pub-sub). 카프카는 **하나의 모델로 둘 다** 표현한다.

### Rebalancing (리밸런싱)

컨슈머가 그룹에 들어오거나 빠지면(배포, 장애, 스케일링), 파티션 할당이 다시 계산된다. 이 과정에서 잠깐 소비가 멈출 수 있어서 운영 시 주의해야 할 포인트다 (4단계 심화).

---

## 6. Broker & Cluster — 서버 쪽

- **Broker**: 카프카 서버 1대. 파티션 데이터를 실제로 디스크에 저장하고 요청을 처리한다.
- **Cluster**: 여러 브로커의 모음. 파티션들이 여러 브로커에 분산 저장된다.
- 과거에는 **ZooKeeper**가 클러스터 메타데이터/리더 선출을 담당했지만, 최신 버전은 **KRaft 모드**로 ZooKeeper 없이 카프카 자체가 처리한다. **새로 배우면 KRaft 기준으로 시작하라.** (ZooKeeper는 레거시)

---

## 7. Replication — 장애에 견디는 법

각 파티션은 안정성을 위해 여러 브로커에 복제된다. `replication factor = 3`이면 같은 파티션의 사본이 3개 브로커에 존재한다.

- **Leader**: 해당 파티션의 모든 읽기/쓰기를 담당하는 사본 1개.
- **Follower**: 리더를 실시간으로 복제하는 나머지 사본들.
- **ISR (In-Sync Replicas)**: 리더와 충분히 동기화된 사본들의 집합. 리더 브로커가 죽으면 ISR 안의 팔로워 중 하나가 새 리더로 승격된다 → 무중단.

```text
Partition 0 (replication factor = 3)
┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│  Broker 1   │   │  Broker 2   │   │  Broker 3   │
│  Leader ★   │ → │  Follower   │   │  Follower   │
│ (read/write)│   │  (복제)      │   │  (복제)      │
└─────────────┘   └─────────────┘   └─────────────┘
   Broker 1이 죽으면 → Broker 2 또는 3이 새 Leader로 승격
```

## 8. 전체 그림 한 장으로

```text
                  ┌─────────────────────────────────────────┐
   Producer  ───► │            Kafka Cluster                │
   (Key로         │                                         │
   파티션 결정)    │   Topic: order-created                  │
                  │   ├ Partition 0 [■■■■■→]  (Broker1 리더) │
                  │   ├ Partition 1 [■■■→]    (Broker2 리더) │
                  │   └ Partition 2 [■■■■■■■→](Broker3 리더) │
                  │        (각 파티션은 다른 브로커에 복제됨)  │
                  └─────────────────────────────────────────┘
                          │                    │
              Consumer Group A         Consumer Group B
              (파티션 나눠 처리)         (전체 독립 소비)
```

## 9. 스스로 점검하는 질문 (이해도 체크)

다음에 즉답할 수 있으면 1단계 완료다:

1. 카프카에서 메시지는 컨슈머가 읽으면 삭제되는가? → **아니오** (보존 정책으로만 삭제)
2. 토픽 전체에서 메시지 순서가 보장되는가? → **아니오** (파티션 내부에서만 보장)
3. 컨슈머를 파티션 수보다 많이 띄우면 더 빨라지는가? → **아니오** (남는 컨슈머는 논다)
4. 같은 토픽을 두 서비스가 각각 전부 소비할 수 있는가? → **예** (서로 다른 컨슈머 그룹)
5. "같은 주문의 이벤트는 순서대로 처리"를 보장하려면? → **주문 ID를 메시지 Key로** 사용
6. 처리했던 과거 메시지를 다시 읽을 수 있는가? → **예** (오프셋을 되감으면 됨)
7. 읽은 위치(오프셋)는 누가 관리하는가? → **컨슈머** (브로커가 아님)

## 10. 핵심 용어 한눈에

| 용어 | 한 줄 정의 |
|---|---|
| Topic | 메시지가 쌓이는 논리적 카테고리 (DB의 테이블) |
| Partition | 토픽을 쪼갠 물리적 로그. 병렬성·순서의 단위 |
| Offset | 파티션 내 메시지 위치(0부터). 컨슈머의 책갈피 |
| Producer | 토픽에 메시지를 쓰는 주체 |
| Consumer | 토픽에서 메시지를 읽는 주체 |
| Consumer Group | 파티션을 나눠 갖는 컨슈머들의 집합 |
| Consumer Lag | 최신 오프셋 − 커밋 오프셋. 밀린 정도 |
| Broker | 카프카 서버 1대 |
| Cluster | 브로커들의 모음 |
| Replication Factor | 파티션 사본 개수 |
| Leader / Follower | 읽기쓰기 담당 사본 / 복제 사본 |
| ISR | 리더와 동기화된 사본 집합 (리더 후보) |
| KRaft | ZooKeeper를 대체한 메타데이터 관리 모드 (최신) |
| Rebalancing | 컨슈머 증감 시 파티션 재할당 |
