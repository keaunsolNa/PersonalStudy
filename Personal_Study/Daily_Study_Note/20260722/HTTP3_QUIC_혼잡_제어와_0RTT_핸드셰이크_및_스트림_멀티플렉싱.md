# HTTP/3 QUIC 혼잡 제어와 0-RTT 핸드셰이크 및 스트림 멀티플렉싱

> 2026-07-22 신규 주제 · 확장 대상: Network/CS(전송 계층·프로토콜)

## 학습 목표

- QUIC이 UDP 위에서 신뢰성·순서·혼잡 제어를 재구현한 배경과 TCP HOL 블로킹 문제를 분석한다.
- QUIC 스트림 멀티플렉싱이 HTTP/2의 전송 계층 HOL 블로킹을 어떻게 제거하는지 설명한다.
- TLS 1.3 통합 핸드셰이크와 1-RTT·0-RTT 연결 수립 및 재전송 공격 위험을 파악한다.
- Connection ID 기반 연결 마이그레이션과 QUIC 혼잡 제어 신호의 정밀도를 비교한다.

## 1. QUIC의 출발점 — TCP의 구조적 한계

HTTP/2는 하나의 TCP 연결에서 여러 스트림을 멀티플렉싱해 HTTP/1.1의 애플리케이션 계층 HOL(Head-of-Line) 블로킹을 없앀다. 그러나 **전송 계층 HOL 블로킹**은 남았다. TCP는 바이트 스트림을 순서대로 전달하므로, 한 패킷이 유실되면 그 뒤에 도착한 모든 패킷이 재전송을 기다리며 커널 버퍼에 묶인다. HTTP/2가 논리적으로 독립인 스트림 10개를 한 TCP 연결에 실었어도, 스트림 3의 패킷 하나가 유실되면 스트림 5·7의 데이터까지 애플리케이션에 전달되지 못한다. 손실률이 높은 모바일 네트워크에서 이 문제가 두드러진다.

QUIC은 이를 근본적으로 해결하기 위해 TCP 대신 **UDP 위에 신뢰성 전송을 새로 구현**한다. UDP는 순서·재전송을 보장하지 않는 얇은 계층이라, QUIC이 스트림별로 독립적인 순서·재전송을 관리할 자유를 얻는다. QUIC은 커널이 아니라 **유저스페이스**에 구현되므로 OS 업데이트 없이 프로토콜을 빠르게 진화시킬 수 있다는 부수 효과도 크다. HTTP/3는 이 QUIC 위에서 동작하는 HTTP 매핑이다.

## 2. 스트림 멀티플렉싱 — 독립적 순서 보장

QUIC의 핵심은 각 스트림이 **독립적인 순서 공간**을 갖는다는 것이다. 스트림 A의 패킷 유실은 스트림 A의 후속 데이터만 지연시키고, 스트림 B·C는 영향받지 않는다. 애플리케이션은 손실 없는 스트림의 데이터를 즉시 받을 수 있다.

```
QUIC 패킷 = [헤더][여러 프레임]
프레임 예: STREAM(id=3, offset=1024, data=...)
          STREAM(id=5, offset=0, data=...)
          ACK(...)

스트림 3 의 offset 512~1023 패킷이 유실되어도
스트림 5 의 데이터는 애플리케이션에 정상 전달됨 (독립 순서 공간)
```

각 STREAM 프레임은 스트림 ID와 offset을 명시한다. 수신 측은 스트림별로 offset을 조립하므로, 한 스트림의 구멍이 다른 스트림 조립을 막지 않는다. 이것이 TCP 단일 바이트 스트림과의 결정적 차이다. TCP는 연결 전체가 하나의 순서 공간이라 어느 패킷 유실이든 전체를 막지만, QUIC은 유실 영향을 해당 스트림으로 격리한다. 그 결과 HTTP/3는 손실 환경에서 HTTP/2보다 체감 지연이 낮다.

## 3. TLS 1.3 통합과 1-RTT 핸드셰이크

TCP+TLS 조합은 핸드셰이크가 분리돼 있다. TCP 3-way 핸드셰이크(1 RTT)를 마친 뒤 TLS 핸드셰이크(TLS 1.3에서 1 RTT)를 하므로 데이터 전송까지 최소 2 RTT가 든다. QUIC은 전송과 암호화 핸드셰이크를 **하나로 통합**해 1 RTT만에 암호화된 데이터 전송을 시작한다. QUIC은 TLS 1.3을 필수로 내장하며, 암호화되지 않은 QUIC은 존재하지 않는다.

```
TCP + TLS 1.3:
  Client → SYN            (RTT 1 시작)
  Server → SYN-ACK
  Client → ACK
  Client → TLS ClientHello (RTT 2 시작)
  Server → TLS ServerHello + 인증
  Client → 암호화 데이터 전송 가능    → 총 2 RTT

QUIC (1-RTT):
  Client → Initial(ClientHello 포함)  (RTT 1 시작)
  Server → Initial+Handshake(ServerHello, 인증)
  Client → 암호화 데이터 전송 가능    → 총 1 RTT
```

첫 패킷(Initial)에 TLS ClientHello를 실어 보내므로 전송 핸드셰이크와 키 교환이 동시에 진행된다. 헤더 일부(패킷 번호 등)까지 암호화·인증하므로 중간 장비가 프로토콜 내부를 관찰·조작하기 어렵다. 이 "헤더 암호화"는 프로토콜 경직화(ossification)를 막아 향후 진화 여지를 남기는 설계 의도도 있다.

## 4. 0-RTT — 재연결 시 즉시 전송

이전에 연결한 적 있는 서버라면 QUIC은 **0-RTT**로 첫 왕복에 애플리케이션 데이터를 실어 보낼 수 있다. 이전 세션에서 받은 세션 티켓(pre-shared key)으로 첫 패킷부터 데이터를 암호화한다. 재방문이 잦은 웹에서 지연을 크게 줄인다.

```
QUIC 0-RTT (재연결):
  Client → Initial(0-RTT 데이터 포함: GET /page)  (RTT 0 에 요청 전송!)
  Server → 응답 시작
```

그러나 0-RTT에는 **재전송 공격(replay attack)** 위험이 있다. 0-RTT 데이터는 서버가 아직 완전한 핸드셰이크로 신선도(freshness)를 검증하기 전에 처리되므로, 공격자가 캐프처한 0-RTT 패킷을 재전송하면 같은 요청이 중복 실행될 수 있다. 그래서 **0-RTT는 멱등 요청(GET 등 안전한 메서드)에만** 써야 한다. 상태를 변경하는 POST·결제 같은 요청은 0-RTT로 보내면 안 된다. 실무에서는 CDN·서버가 0-RTT를 GET류에만 허용하고, 변경 요청은 1-RTT 완료 후로 뮸룬는 정책을 둔다. 이 멱등성 제약이 0-RTT를 쓸 때 반드시 기억할 함정이다.

## 5. 혼잡 제어 — 정밀한 손실 신호

QUIC의 혼잡 제어 알고리즘은 TCP와 유사한 계열(NewReno, CUBIC, BBR)을 쓰지만, **신호 정밀도**가 다르다. TCP의 재전송 모호성(retransmission ambiguity) 문제 — 재전송된 세그먼트의 ACK가 원본에 대한 것인지 재전송분에 대한 것인지 구분 못하는 문제 — 를 QUIC은 **단조 증가하는 패킷 번호**로 해결한다. QUIC은 재전송해도 새 패킷 번호를 부여하므로 모든 ACK가 정확히 어느 전송에 대한 것인지 특정된다.

```
TCP: 세그먼트 seq=1000 유실 → 재전송도 seq=1000
     → ACK 1000 이 원본용인지 재전송용인지 모호 → RTT 추정 부정확

QUIC: 패킷 번호 42 유실 → 재전송은 새 번호 57 로 (같은 데이터, 다른 번호)
      → ACK 57 은 명확히 재전송분 → 정확한 RTT 샘플 → 정밀 혼잡 제어
```

또 QUIC ACK 프레임은 여러 개의 ACK 범위를 담을 수 있어 TCP SACK보다 표현력이 크고, 수신 측이 어떤 패킷을 받았는지 세밀히 알린다. 이 정밀도가 손실·재정렬이 잦은 네트워크에서 더 나은 대역폭 추정과 빠른 손실 복구를 가능케 한다. 혼잡 제어 로직이 유저스페이스에 있어 서버가 BBR 같은 알고리즘을 자유롭게 배포·튜닝할 수 있는 것도 장점이다.

## 6. Connection ID와 연결 마이그레이션

TCP 연결은 (출발지 IP·포트, 목적지 IP·포트) 4-튜플로 식별된다. 모바일 기기가 Wi-Fi에서 셀룰러로 전환하면 IP가 바뀌어 TCP 연결이 끊기고 재수립해야 한다. QUIC은 연결을 4-튜플이 아니라 **Connection ID**로 식별한다. 네트워크가 바뀜 그 IP·포트가 변해도 Connection ID가 같으면 서버는 같은 연결로 인식해 이어간다.

```
QUIC 패킷 헤더에 Connection ID 포함
  Wi-Fi (IP a.a.a.a) → 셀룰러 (IP b.b.b.b) 전환
  → 4-튜플은 변했지만 Connection ID 동일
  → 서버가 같은 연결로 처리 → 핸드셰이크 재수립 불필요 (연결 마이그레이션)
```

이 연결 마이그레이션은 모바일 환경에서 끊김 없는 스트리밍·다운로드를 가능케 한다. 다만 Connection ID가 고정이면 경로를 넘나드는 사용자를 추적하는 프라이버시 문제가 생긴다. QUIC은 이를 위해 여러 개의 Connection ID를 미리 발급하고 경로 변경 시 새 ID로 교체(rotation)하는 메커니즘을 둔다. 서버와 클라이언트가 사전에 여분의 Connection ID를 교환해 두었다가 마이그레이션 시 아직 관찰되지 않은 ID로 전환해 연결 추적 가능성을 줄인다.

## 7. HTTP/3 매핑과 QPACK

HTTP/3는 QUIC 스트림 위에 HTTP 시맨틱을 엹는다. 각 HTTP 요청·응답은 하나의 양방향 QUIC 스트림에 실린다. HTTP/2의 헤더 압축 HPACK은 단일 순서 스트림을 전제로 동적 테이블을 관리했는데, QUIC은 스트림이 독립·비순서라 HPACK을 그대로 쓸 수 없다. 그래서 HTTP/3는 **QPACK**을 쓴다.

QPACK은 헤더 압축의 동적 테이블 업데이트를 별도의 단방향 스트림(encoder/decoder 스트림)로 분리해, 요청 스트림 간 순서 의존성을 끊는다. 다만 동적 테이블 참조가 아직 도착하지 않은 테이블 갱신에 의존하면 잠시 블로킹될 수 있어, QPACK은 "얼마나 최근 항목까지 참조할지"를 조절하는 blocked-stream 한계를 둔다. 압축률과 HOL 블로킹 위험 사이의 절충을 파라미터로 노출하는 것이다. 실무 서버는 이 값을 보수적으로 잡아 압축을 조금 포기하고 블로킹을 피하는 쪽을 택하는 경우가 많다.

## 8. 도입 trade-off와 현실

QUIC/HTTP3의 이점은 명확하지만 비용도 있다. 첫째, **CPU 부담**이다. TCP는 커널·NIC 하드웨어 오프로드(TSO, GRO)의 수십 년 최적화를 누리지만, QUIC은 유저스페이스에서 패킷마다 암호화·혼잡 제어를 수행해 같은 처리량에 더 많은 CPU를 쓴다. 대규모 서버는 이 오버헤드가 비용으로 직결된다. 둘째, **UDP 차단·성능 저하**다. 일부 방화벽·미들박스가 UDP를 제한하거나 QoS에서 낮게 취급해, 클라이언트는 QUIC 실패 시 TCP로 폴백하는 경로를 반드시 갖춰야 한다.

그래서 현실의 브라우저·CDN은 HTTP/3를 우선 시도하되(`Alt-Svc` 헤더로 광고), 실패하면 HTTP/2(TCP)로 우아하게 폴백한다. 손실이 큰 모바일·장거리 경로에서 QUIC의 이득이 크고, 손실이 거의 없는 데이터센터 내부 통신에서는 TCP 대비 이득이 작거나 CPU 비용이 앞선다. 결국 QUIC은 "고손실·고지연·이동성 큰 라스트마일"에서 빛나고, 그 조건이 약한 곳에서는 도입 판단을 신중히 해야 하는 프로토콜이다.

## 9. 배포 실무 — Alt-Svc, 폴백, 서버 설정

HTTP/3는 별도 포트를 새로 뚫는 대신 기존 HTTPS(TCP/443)로 접속한 클라이언트에게 "나는 UDP/443에서 HTTP/3도 지원한다"고 알린다. 이 광고가 `Alt-Svc`(Alternative Services) 응답 헤더다.

```
HTTP/2 200
alt-svc: h3=":443"; ma=86400
  # h3 를 UDP 443 에서 제공, 이 정보를 86400초(1일) 캐시하라
```

브라우저는 첫 요청을 HTTP/2(TCP)로 받고 `Alt-Svc`를 본 뒤, 이후 요청부터 HTTP/3(QUIC)를 시도한다. QUIC 연결이 실패하거나 UDP가 차단되면 다시 TCP로 폴백한다. 이 "happy eyeballs" 방식 덕분에 HTTP/3 미지원·차단 환경에서도 서비스가 끊기지 않는다. 서버 측에서는 Nginx(1.25+), Caddy, Cloudflare 같은 프록시가 QUIC 리스너와 `Alt-Svc` 광고를 함께 설정한다.

```nginx
server {
    listen 443 ssl;         # TCP (HTTP/2)
    listen 443 quic reuseport;   # UDP (HTTP/3)
    ssl_protocols TLSv1.3;       # QUIC 은 TLS 1.3 필수
    add_header Alt-Svc 'h3=":443"; ma=86400';
}
```

방화벽에서 UDP/443 인바운드를 반드시 열어야 하며, 로드밸런서가 UDP를 지원하고 같은 Connection ID를 같은 백엔드로 라우팅(연결 마이그레이션 유지)하도록 구성해야 한다. 또 QUIC의 UDP 패킷은 경로 MTU를 넘으면 단편화·유실되기 쉬우므로, 서버는 보수적 초기 데이터그램 크기(약 1200바이트)를 쓰고 PMTU 탐색으로 점진적으로 키운다. 이 설정들을 놓치면 "브라우저는 HTTP/3를 시도했다가 실패해 매번 TCP로 폴백"하는, 이점 없이 왕복만 늘어나는 상태가 된다. 배포 후에는 실제로 h3로 협상되는 비율을 로그·`curl --http3`로 검증하는 것이 필수다.

한 가지 더 고려할 것은 **앱플리피케이션 공격 방어**다. QUIC 서버는 아직 검증되지 않은 클라이언트 주소로 과도한 데이터를 보내면 DDoS 반사 공격에 악용될 수 있다. 그래서 QUIC은 주소 검증 전에는 서버가 받은 바이트의 3배까지만 보낼 수 있도록 제한한다(anti-amplification limit). 핸드셰이크 초기에 서버가 Retry 패킷으로 클라이언트 주소 소유를 확인하는 절차도 이 방어의 일부다. 운영자는 이 메커니즘이 켜져 있는지, 그리고 초기 혼잡 윈도우가 이 3배 제한과 상호작용해 첫 왕복 처리량이 제약되지 않는지 점검한다. 이처럼 QUIC의 보안·성능 파라미터는 서로 얘혀 있어, 하나를 공격 방어를 위해 조이면 다른 하나(초기 대역폭)가 영향받는 절충 관계에 있다.
