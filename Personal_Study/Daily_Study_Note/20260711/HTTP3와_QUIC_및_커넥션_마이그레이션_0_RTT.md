# HTTP/3와 QUIC 및 커넥션 마이그레이션 0-RTT

> 2026-07-11 신규 주제 · 확장 대상: 네트워크 (TLS 1.3·TCP 혼잡제어·HTTP2 학습됨)

## 학습 목표

- QUIC이 UDP 위에서 신뢰성·혼잡제어·암호화를 재구성한 이유를 설명한다
- 스트림 다중화가 TCP의 HOL 블로킹을 어떻게 제거하는지 분해한다
- 커넥션 ID 기반 마이그레이션과 0-RTT 재개의 동작과 위험을 판단한다
- HTTP/3의 QPACK 헤더 압축과 HTTP/2 HPACK의 차이를 구분한다

## 1. QUIC의 등장 배경

HTTP/2는 하나의 TCP 커넥션에서 여러 스트림을 다중화해 HTTP/1.1의 커넥션당 직렬 처리 문제를 풀었다. 그러나 TCP 계층에서 패킷 하나가 유실되면 그 뒤 모든 데이터가 재전송을 기다리며 멈춘다. 스트림은 논리적으로 독립이지만 TCP는 바이트 스트림을 순서대로만 전달하므로, 한 스트림의 손실이 무관한 다른 스트림까지 막는 전송 계층 HOL(Head-of-Line) 블로킹이 남았다.

QUIC은 이 문제를 전송 계층에서 근본적으로 해결하기 위해 UDP 위에 신뢰성·흐름 제어·혼잡 제어·암호화를 새로 구현한 프로토콜이다. 커널의 TCP 스택을 우회하고 사용자 공간에서 동작하므로, OS 업데이트 없이 프로토콜을 진화시킬 수 있다는 배포상의 이점도 크다. HTTP/3는 이 QUIC 위에서 동작하는 HTTP 매핑이다.

QUIC은 TLS 1.3을 프로토콜에 내장한다. TLS를 별도 계층으로 얹는 대신 핸드셰이크를 QUIC 패킷에 통합해, 연결 수립과 암호화 협상을 하나의 왕복으로 합쳤다. 앞서 학습한 TLS 1.3의 1-RTT 핸드셰이크와 0-RTT 재개가 QUIC에서 그대로 활용된다.

## 2. 스트림 다중화와 HOL 블로킹 제거

QUIC은 스트림을 프로토콜의 일급 개념으로 다룬다. 각 스트림은 독립적인 흐름 제어와 순서 보장을 가지며, 서로 다른 스트림의 데이터는 서로의 순서에 영향을 주지 않는다. 한 스트림의 패킷이 유실되면 그 스트림만 대기하고 다른 스트림은 계속 진행한다. 전송 계층 HOL 블로킹이 사라지는 것이다.

```
TCP(HTTP/2): [stream1-a][stream2-a][stream1-b(손실)][stream2-b]
             → stream1-b 재전송 대기 중 stream2-b도 블록

QUIC(HTTP/3): stream1: [a][b(손실)] → stream1만 대기
              stream2: [a][b]       → 정상 전달
```

스트림은 단방향·양방향으로 나뉘고, 클라이언트/서버가 개시하는 스트림에 짝수·홀수 ID를 부여해 충돌 없이 생성한다. 흐름 제어는 스트림 단위와 커넥션 단위로 이중 적용되어, 한 스트림이 수신 버퍼를 독점하지 못하게 한다.

## 3. 커넥션 수립과 패킷 구조

QUIC 커넥션은 핸드셰이크로 시작한다. 클라이언트가 Initial 패킷에 TLS ClientHello를 실어 보내면, 서버가 ServerHello와 인증서를 응답한다. 신규 연결은 1-RTT 만에 애플리케이션 데이터를 보낼 수 있어 TCP+TLS의 최소 2-RTT(TCP 3-way 1-RTT + TLS 1-RTT)보다 빠르다.

QUIC 패킷은 롱 헤더(핸드셰이크용)와 숏 헤더(연결 수립 후)로 나뉜다. 패킷 번호는 단조 증가하며 암호화되어, 재전송된 프레임도 새 패킷 번호로 실려 나간다. 이 점이 TCP와 크게 다르다. TCP는 시퀀스 번호를 재사용해 원본과 재전송을 구분하지 못하는 재전송 모호성(retransmission ambiguity)이 있어 RTT 측정이 부정확했는데, QUIC은 패킷 번호가 항상 새 값이라 정확한 RTT 샘플을 얻는다.

패킷 안에는 여러 프레임이 담긴다. STREAM 프레임(데이터), ACK 프레임(수신 확인), CRYPTO 프레임(핸드셰이크), MAX_DATA·MAX_STREAM_DATA(흐름 제어) 등이 하나의 패킷에 섞여 효율적으로 전달된다.

## 4. 커넥션 마이그레이션

TCP 연결은 4-튜플(출발지/목적지 IP·포트)로 식별된다. 그래서 모바일 기기가 Wi-Fi에서 셀룰러로 전환하며 IP가 바뀌면 연결이 끊기고 재수립해야 한다. QUIC은 연결을 IP가 아니라 커넥션 ID(Connection ID)로 식별한다. 경로가 바뀌어도 커넥션 ID가 같으면 같은 연결로 인식해, 진행 중인 스트림을 끊김 없이 이어간다.

```
Wi-Fi(IP-A) ──QUIC conn(CID=0xAB)──> 서버
   ↓ 네트워크 전환 (IP 변경)
셀룰러(IP-B) ──QUIC conn(CID=0xAB)──> 서버 (같은 연결 유지)
```

마이그레이션 시 서버는 새 경로가 정말 그 클라이언트인지 검증해야 한다. 공격자가 커넥션 ID를 가로채 트래픽을 자기 쪽으로 돌리는 것을 막기 위해, 서버는 새 경로로 PATH_CHALLENGE 프레임을 보내고 클라이언트가 PATH_RESPONSE로 답해야 경로를 확정한다(경로 검증). 또한 프라이버시를 위해 여러 개의 커넥션 ID를 미리 발급해 두고 경로 전환 시 새 ID로 바꿔, 네트워크 관찰자가 IP 변경 전후를 연결짓지 못하게 한다.

## 5. 0-RTT 재개와 위험

QUIC은 TLS 1.3의 0-RTT를 활용해, 이전에 연결했던 서버에는 핸드셰이크 완료 전에 애플리케이션 데이터를 보낼 수 있다. 클라이언트는 이전 세션에서 받은 재개 토큰과 PSK로 첫 패킷에 데이터를 실어, 왕복 없이 요청을 시작한다. 반복 방문 시 지연이 크게 준다.

```
1-RTT: ClientHello ──> ServerHello ──> [데이터]
0-RTT: ClientHello + [early data] ──> ServerHello + 응답
```

그러나 0-RTT 데이터는 재전송 공격(replay)에 취약하다. 공격자가 0-RTT 패킷을 캡처해 재전송하면 서버가 같은 요청을 두 번 처리할 수 있다. 핸드셰이크가 아직 완료되지 않아 서버가 신선도(freshness)를 보장하지 못하기 때문이다. 그래서 0-RTT에는 멱등(idempotent) 요청만 실어야 한다. GET처럼 상태를 바꾸지 않는 요청은 안전하지만, 결제·주문 같은 비멱등 요청을 0-RTT로 보내면 이중 처리 위험이 있다. 실무에서는 서버가 0-RTT에서 비멱등 메서드를 거부하고 1-RTT로 재시도하게 강제한다.

## 6. QPACK 헤더 압축

HTTP/2는 HPACK으로 헤더를 압축했다. HPACK은 동적 테이블을 순서대로 갱신해 이전 헤더를 참조하는데, 이 순서 의존성이 QUIC의 독립 스트림과 충돌한다. QUIC에서 스트림이 순서 없이 도착하면 HPACK 동적 테이블의 일관성이 깨진다. 그래서 HTTP/3는 QPACK이라는 변형을 쓴다.

QPACK은 헤더 압축 지시를 두 개의 단방향 스트림(encoder/decoder stream)으로 분리하고, 동적 테이블 참조가 아직 도착하지 않은 삽입을 가리키지 않도록 "blocked" 처리를 도입한다. 인코더는 참조하는 테이블 항목이 디코더에 반영됐음을 보장하는 수준에서만 동적 참조를 쓰고, 그렇지 않으면 리터럴로 보내 HOL 블로킹을 피한다. `SETTINGS_QPACK_MAX_BLOCKED_STREAMS`로 블록 허용 개수를 조절해 압축률과 지연 사이를 절충한다.

## 7. 트레이드오프와 배포 현실

| 항목 | TCP+TLS(HTTP/2) | QUIC(HTTP/3) |
|------|------------------|--------------|
| 전송 계층 HOL | 존재 | 제거 |
| 최초 연결 | 2-RTT | 1-RTT |
| 재방문 | TLS 0-RTT 가능 | 0-RTT 가능 |
| IP 변경 | 재연결 | 마이그레이션 |
| 구현 위치 | 커널 | 사용자 공간 |
| CPU 비용 | 낮음 | 상대적으로 높음 |

QUIC의 사용자 공간 구현은 유연하지만 CPU 비용이 크다. 커널의 TCP는 오랜 세월 최적화되고 하드웨어 오프로딩(TSO/GRO)을 받지만, UDP 기반 QUIC은 패킷마다 사용자 공간에서 암호화·혼잡 제어를 처리해 높은 처리량에서 CPU를 더 쓴다. UDP GSO 같은 오프로딩이 도입되며 격차가 줄고 있으나 여전히 고려 요소다.

또 다른 현실은 UDP 차단이다. 일부 기업 방화벽·미들박스가 UDP를 제한해 QUIC이 막힐 수 있다. 그래서 클라이언트는 QUIC을 시도하되 실패하면 TCP HTTP/2로 폴백하는 이중 경로를 갖는다. 서버는 `Alt-Svc` 헤더로 HTTP/3 가용성을 광고하고, 클라이언트가 다음 연결에서 QUIC을 시도하게 유도한다.

## 8. 실전 관점

앞서 학습한 TCP 혼잡 제어(CUBIC·BBR)는 QUIC에도 그대로 적용된다. QUIC은 혼잡 제어 알고리즘을 프로토콜에 고정하지 않아 BBR 같은 최신 알고리즘을 사용자 공간에서 자유롭게 교체할 수 있다. 이는 커널 업그레이드가 필요한 TCP보다 실험·배포가 빠르다는 장점으로 이어진다.

서비스 관점에서 HTTP/3 도입은 대개 CDN·엣지 계층에서 종단되고, 오리진과의 통신은 여전히 TCP인 경우가 많다. 사용자와 가까운 엣지에서 QUIC의 지연·마이그레이션 이점을 취하고, 데이터센터 내부는 안정적인 TCP를 쓰는 하이브리드가 일반적이다. 0-RTT는 캐시 조회처럼 멱등한 엣지 요청에 적용하고 상태 변경 요청은 배제하는 정책을 CDN 설정에서 명시해야 재전송 위험을 통제할 수 있다.

모니터링은 TCP와 다른 지표가 필요하다. 커넥션 마이그레이션 빈도, 0-RTT 수락·거부율, QUIC 핸드셰이크 실패 후 TCP 폴백율을 관측해야 사용자 경험을 파악할 수 있다. 전송 계층이 사용자 공간으로 옮겨 온 만큼 애플리케이션이 얻는 관측 정보도 풍부해진다.
